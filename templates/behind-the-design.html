Behind the Design

A short tour of how this app is built and the reasoning behind each choice.

The stack

This is a FastAPI app serving Jinja templates, backed by a PostgreSQL database, with charts rendered in Plotly. It's deployed on Railway and refreshes its data once a day.

I chose FastAPI because my background is in Python and data work. It lets me stand up a clean web layer with very little ceremony, so I can spend my time where my strengths and interests are — the backend and the data — rather than on heavy frontend tooling. The templating layer handles the presentation, which frees me to focus on the pipeline that feeds it.

Why Postgres

The data here is tiny — one runner's activities, a few hundred rows growing by roughly one per day. At that scale the classic "analytical vs. transactional database" debate barely applies; any engine returns queries instantly. So the deciding factors weren't raw analytical horsepower — they were fit for a deployed app:

  • It's a client-server database, which is what a 24/7 app needs. The web app and the daily sync job are two separate processes that both touch the same data — one reading, one writing. Postgres handles that concurrent access natively.
  • It decouples storage from the app. On a cloud host the container's filesystem is ephemeral and gets wiped on every deploy. A managed Postgres lives independently, so the data survives deploys and the app stays stateless.
  • It's right-sized. Reaching for a large analytical warehouse to serve a few hundred rows would be over-engineering. Matching the tool to the actual workload is the point.

I started with DuckDB, which I love for local, exploratory analysis — it's excellent for blending SQL and Python in a notebook. But it's an embedded, single-writer, file-based engine: great in a notebook, awkward as a shared serving layer behind a live app. So I kept DuckDB for exploration and moved the serving database to Postgres.

The pipeline

The ETL is deliberately simple: a daily incremental insert, scheduled with cron.

Each morning the job authenticates with the Strava API, requests only activities newer than the most recent one already stored (a watermark), and inserts them. The insert is idempotent — it uses ON CONFLICT DO NOTHING on the activity ID as a primary key — so re-runs or overlapping windows can never create duplicates. Raw API data lands in a staging table, then a single transformation query curates it into the typed, unit-converted table the app reads (a small bronze → silver pattern).

Importantly, the web app reads only from the database, never live from Strava. Page loads are fast and never depend on the API being up or on rate limits — the cron job is the only thing that talks to Strava.

What I'd change at scale

The design is scoped to what this project actually is: one athlete, small data, one machine. If it were built for many Strava users, the choices would shift:

  • Data warehouse: I'd move analytics to BigQuery (or a similar columnar warehouse), which is built for scanning large volumes across many users.
  • Orchestration: a daily cron script is perfect for a single, simple job. For more complex, multi-step, dependency-aware pipelines I'd reach for Airflow — real scheduling, retries, backfills, and monitoring.
