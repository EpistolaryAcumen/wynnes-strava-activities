Behind the Design

The app is indeed simple, but each brick in this house is the foundation of a data life-cycle.

The Stack

This is a FastAPI app serving Jinja templates, backed by a Postgres database. The charts are rendered in Plotly and the thing is deployed on Railway, refreshed hourly.

I chose FastAPI because it keeps me in my comfort-zone, Python, SQL and data. The templating layer handles the presentation which frees me up to focus on the pipeline that feeds it.

The Pipeline

The ETL is deliberately simple: an hourly incremental insert, scheduled with cron.

Each morning the job authenticates with the Strava API, requests only activities newer than the most recent one already stored, and inserts them. The insert is idempotent. It uses ON CONFLICT DO NOTHING on the activity ID as a primary key ensuring that re-runs or overlapping windows can never create duplicates.

The web app reads only from the database, never live from Strava. Page loads are fast and never depend on the API being up or on rate limits. The cron job is the only thing that talks to Strava.

What I'd do at scale

If it was built for many users, my design choices would shift accordingly:

  • Data warehouse: I'd use BigQuery. Its columnar and built for scanning large volumes across many users.
  • Orchestration: For more complex, dependency-aware pipelines I'd use Airflow as it is great for scheduling, retries, backfills, and monitoring.
